1. Introduction

Flash Code is a VS Code extension that turns a chat message into real work: it understands intent, reads your codebase, edits files (with reviewable diffs), runs commands (with approval), delegates to specialized subagents, and keeps going until the objective is genuinely complete. It talks to eight AI providers through one pluggable engine and survives Google Gemini's free tier with multi-key round-robin rotation.

This guide is written for contributors. It assumes you can read TypeScript and know the basics of the VS Code extension API. Three design commitments shape everything:

① Host-agnostic engine

Everything under src/core/ and src/providers/ is forbidden from importing vscode. Host capabilities (filesystem, shell, UI) are injected through the ToolContext interface. This is what makes the engine unit-testable with fakes and conceptually portable.

② One source of truth per concern

Tools are declared once and projected to both native function-calling and XML fallback. Providers map to/from one canonical StreamEvent stream. Prompts assemble from composable constants. No duplicated, drift-prone definitions.

③ Preserve the original UI

The webview is the original hand-authored Flash Code UI (vanilla HTML/JS). The new engine drives it through a host adapter rather than replacing it — the look and feel is intentional and protected.

6. Triage & Routing

Routing is what stops "how do circuit breakers work?" from crawling your files, and what sends "fix this bug" or "search the web for X" into the full tool loop. It is a single cheap classification call (core/triage.ts + prompts/triage.ts).

Figure 6.1 — Intent routing decision tree.
RouteToolsIteration capExamples
generalnone (direct answer)—"what is a closure", "explain async/await"
codebaseREAD_ONLY_TOOLS12"explain this codebase", "where is X handled?"
agenticall tools40"build a todo app", "search the web for…", "use your WebScout agent"
The decisive rule baked into TRIAGE_PROMPT: if answering needs the web, a URL, live info, a subagent, or any tool/action → agentic; only choose general if it's answerable from the model's own knowledge.

7. The Agent Loop

core/agentLoop.ts is the heart of the engine. It is provider-agnostic: native tool-use and the XML fallback both reduce to the same { text, thinking, toolCalls, finish } shape via the adapter.

flowchart TD
  S(["run(userText, history)"]) --> I{"iter < maxIterations?"}
  I -->|no| DONE(["emit done · return finalProse"])
  I -->|yes| TURN["adapter.runTurn → stream a turn"]
  TURN --> TRUNC{"truncated XML tag?"}
  TRUNC -->|yes| NUDGE1["ask to continue"] --> I
  TRUNC -->|no| HASTOOLS{"tool calls?"}
  HASTOOLS -->|no| GATE{"completion gate:
isComplete()?"}
  GATE -->|"DONE or cap"| DONE
  GATE -->|"CONTINUE"| NUDGE2["push 'keep working'"] --> I
  HASTOOLS -->|yes| EXEC["executeTools()
planGate · dedup · spawn-block"]
  EXEC --> PROG{"made progress?"}
  PROG -->|yes| I
  PROG -->|"no (dup calls)"| STALL{"stall++"}
  STALL -->|"1st"| NUDGE3["'stop repeating, answer now'"] --> I
  STALL -->|"2nd"| FORCE["force final answer (no tools)"] --> DONE
Figure 7.1 — The agent loop with its three independent termination guards.

The three guards against infinite loops

GuardMechanismBound
Iteration capmaxIterations12 (codebase) / 40 (agentic)
Stall / dedupA seen set of read-only call signatures; a turn that only repeats prior calls "makes no progress" → one nudge, then a forced final answer.2 stalls
Completion capThe completion gate may push "keep going" only N times.3 continuations

executeTools

For each requested call: resolve the ToolDef; enforce the plan-mode mutation ban (planGate) and the spawn-blocking rule (after a spawn_agent, other tools that turn are ignored); run the handler; emit tool_start/tool_result (output cleaned by toolPreview, which strips the [tool args] label so the card shows real output); feed results back as native tool messages or an XML [tool results] block.

8. The Completion Gate

The agent used to stop the instant a turn had no tool call — even when the prose was a mere intention ("let me check package.json…"). The completion gate fixes that.

stateDiagram-v2
  [*] --> Working
  Working --> WantsToStop: turn has no tool calls
  WantsToStop --> Judge: continuations < 3 ?
  WantsToStop --> Stop: continuations exhausted
  Judge --> Stop: COMPLETION_JUDGE = DONE
  Judge --> Working: = CONTINUE (push "keep working", ++)
  Judge --> Stop: judge call failed (fail-safe)
  Stop --> [*]
Figure 8.1 — Completion-gate state machine (enabled for codebase & agentic routes).

The judge is one cheap call with COMPLETION_JUDGE (temperature 0, ~8 tokens) that reads the original request + the latest message and replies DONE or CONTINUE. It defaults to DONE on any failure, so it can never block or hang. Combined with the loop's other two guards, premature stops are eliminated and infinite loops are impossible.

9. Tool Calling (Native ⇄ XML)

Providers differ: Claude/OpenAI/Gemini support native function calling; local models (Ollama) don't. core/toolCallingAdapter.ts hides this difference so the loop is uniform.

flowchart LR
  R["UnifiedRequest"] --> CAP{"provider.capabilities.tools?"}
  CAP -->|true| N["NATIVE
pass JSON tool schemas
read structured tool_call events"]
  CAP -->|false| Xm["FALLBACK
inject XML tool block in system prompt
parse tags from text (xmlToolParser)"]
  N --> NORM["normalized: {text, thinking, toolCalls, finish}"]
  Xm --> NORM
  NORM --> LOOP["AgentLoop consumes one shape"]
Figure 9.1 — One agent loop, two transports.

Both transports are generated from the same ToolRegistry: schemas() emits native JSON-Schema tools; xmlDefinitions() emits the XML block. They cannot drift because there is one definition per tool.

KeyPool rotation

This is what keeps the Gemini free tier alive. providers/keyPool.ts leases keys round-robin, balanced by in-flight count, paced under an RPM cap, and reroutes on failure.

flowchart TD
  REQ["provider.stream()"] --> ACQ["acquire() — pick key:
fewest in-flight, not cooling down, RPM ok"]
  ACQ --> FETCH["safeFetch + stream"]
  FETCH --> OK{"outcome?"}
  OK -->|200 stream| EMIT["yield StreamEvents"]
  OK -->|"429 RateLimitError"| COOL["cooldown this key → reroute"] --> ACQ
  OK -->|"401/403 AuthError"| DIS["disable key 24h → reroute"] --> ACQ
  OK -->|"5xx / network OverloadError"| BACK["backoff → retry (≤ maxAttempts)"] --> ACQ
Figure 11.1 — KeyPool.withRotation: free-tier survival & resilience.

providers/http.ts's safeFetch converts transient network failures (fetch failed, ECONNRESET, …) into a retryable OverloadError, so a blip reroutes instead of aborting the whole run. classifyStatus maps HTTP status → typed error.

14. Sessions & Storage

session/sessionManager.ts holds the UI transcript (ChatTurn[]), the model-facing ChatMessage[], a rolling summary, and per-turn file snapshots for rewind. Persistence is per-workspace via globalState keys (storage.ts).

FunctionRole

sessionEvents is a cross-webview EventEmitter so the sidebar list refreshes the moment the chat panel saves — that's why a new session appears immediately, titled from your first message.

21. Extending Flash Code

Add a provider

  1. Implement Provider in src/providers/<name>.ts — or reuse OpenAICompatibleProvider for any OpenAI-shaped API.
  2. Register it in registry.ts with its own poolFor(id) KeyPool.
  3. Add config (model, base URL) in package.json and a sidebar option.
  4. Keep wire mapping in pure exported functions (toXMessages) and add a mapper test.
// registry.ts — reuse the OpenAI-compatible base for a new provider
this.register(new OpenAICompatibleProvider({
  id: 'myprovider', label: 'My Provider', keyPool: this.poolFor('myprovider'),
  baseUrl: () => 'https://api.example.com/v1',
  model: () => cfg().get('myprovider.model') || 'default-model',
  defaultModel: 'default-model', models: ['default-model'],
}));

Add a tool

Declare one ToolDef in core/tools/*. Both transports are generated from it. Execute external programs via ctx.runExec (argv) and validate paths with safeRelPath.

{
  name: 'count_lines', category: 'read', mutates: false,
  description: 'Count the lines in a workspace file.',
  parameters: { type: 'object', properties: { path: { type: 'string' } }, required: ['path'] },
  handler: async (args, ctx) => {
    const p = safeRelPath(reqStr(args, 'path'));
    const text = await ctx.readFile(p);
    return `[count_lines ${p}] ${text.split('\n').length} lines`;
  },
}

Add a subagent

Add a profile to SUBAGENTS with a focused system prompt and a tools allowlist (empty = all tools).
